# v0.2 Plan

v0.2 should make PyBetterleaks feel like a Python SDK, not only a thin native
loader. The release theme is:

> Programmatic configuration, measured performance, and stronger release
> confidence without breaking the v0.1 API.

## Release Goals

- Keep v0.1 APIs stable: `scan_text`, `scan_dir`, and `betterleaks_version`
  remain the core public surface.
- Add a typed Python configuration API that can generate Betterleaks-compatible
  config files for a scan.
- Preserve the native bridge contract by passing resolved config paths to Go.
- Publish honest benchmark numbers for common Python workflows.
- Improve release confidence with artifact checksums and at least one more
  platform target if practical.

## Non-Goals

- Do not replace Betterleaks config semantics with a separate rules engine.
- Do not add install-time binary downloads.
- Do not require Pydantic or other runtime Python dependencies.
- Do not introduce async or streaming APIs in v0.2 unless they fall out
  naturally from the config work.
- Do not promise musllinux/Alpine wheels yet; keep Alpine as a canary until
  manylinux wheels are boring.

## Public API Proposal

The main addition is a small set of dataclasses:

```python
from pybetterleaks import BetterleaksConfig, Rule, scan_text

config = BetterleaksConfig(
    rules=[
        Rule(
            id="internal-token",
            description="Internal service token",
            regex=r"INTERNAL_[A-Z0-9]{32}",
            tags=["internal", "token"],
        )
    ],
    allowlist_paths=["tests/fixtures/**"],
)

result = scan_text("INTERNAL_ABCDEFGHIJKLMNOP1234567890", config=config)
```

Planned config-facing types:

- `BetterleaksConfig`: top-level typed config model.
- `Rule`: custom rule definition with id, description, regex, tags, and
  optional metadata.
- `Allowlist`: optional structured allowlist model if plain lists become too
  ambiguous.
- `ConfigFormatError`: raised when a Python config cannot be serialized into a
  valid Betterleaks config.

Planned scanner signatures:

```python
def scan_text(
    text: str,
    *,
    config: BetterleaksConfig | None = None,
    config_path: str | PathLike[str] | None = None,
    validation: bool = False,
    redact: bool = True,
    timeout_seconds: float | None = None,
) -> ScanResult: ...

def scan_dir(
    path: str | PathLike[str],
    *,
    config: BetterleaksConfig | None = None,
    config_path: str | PathLike[str] | None = None,
    validation: bool = False,
    redact: bool = True,
    timeout_seconds: float | None = None,
) -> ScanResult: ...
```

Rules:

- Passing both `config` and `config_path` should raise `ValueError`.
- Passing neither keeps v0.1 behavior.
- `config` serializes to a temporary file and the bridge still receives only
  `config_path`.
- Temporary files must be removed after the native call, including on errors.

## Implementation Milestones

### 1. Config Model And Serialization

- Add `python/pybetterleaks/config.py`.
- Export config types from `pybetterleaks.__init__`.
- Use dataclasses and a narrow stdlib-only TOML serializer for the subset of
  Betterleaks config that PyBetterleaks owns.
- Avoid adding a TOML writer dependency unless the supported config surface
  becomes too broad for a careful local serializer.
- Add unit tests for:
  - minimal config serialization
  - custom rule serialization
  - allowlist serialization
  - invalid regex/config validation
  - mutually exclusive `config` and `config_path`

### 2. Scanner Integration

- Update request construction to accept `config`.
- Serialize config into a temporary directory and pass the generated path to the
  existing bridge request.
- Keep the JSON ABI unchanged for v0.2 unless Betterleaks requires new fields.
- Add tests proving temporary config files are cleaned up.
- Add native smoke tests using an in-memory Python config instead of a fixture
  config file.

### 3. Config Coverage Audit

- Map Betterleaks `v1.6.1` config options to Python models.
- Label every option as:
  - supported in v0.2
  - intentionally omitted
  - blocked pending upstream clarity
- Document unsupported options clearly.
- Prefer partial but correct support over broad, lossy support.

### 4. Benchmarks

- Add `benchmarks/` with reproducible fixtures and a small runner.
- Compare:
  - `pybetterleaks.scan_text`
  - `pybetterleaks.scan_dir`
  - Betterleaks CLI subprocess baseline, for maintainers only
- Measure:
  - cold process startup
  - warm repeated scan
  - directory scan over small, medium, and noisy fixture sets
- Update README only with measured numbers.
- Keep benchmark fixtures synthetic so GitHub push protection is not triggered.

### 5. Release Confidence

- Generate SHA256 checksums for release artifacts.
- Consider Linux arm64 wheels if GitHub runner support is practical.
- Keep Windows amd64, macOS x86_64, macOS arm64, and Linux x86_64 as required
  release platforms.
- Add a clean-environment install verification that installs from the built
  wheelhouse and exercises a config object scan.

### 6. Documentation

- Add config API examples to `README.md` and `docs/getting-started.md`.
- Add a dedicated config guide if the API grows beyond one page.
- Add a benchmark page once measurements exist.
- Update `docs/api.md` to include the config module.
- Update `docs/ai-handoff.md` with v0.2 decisions after implementation.

## Acceptance Criteria

v0.2 is ready when:

- `BetterleaksConfig` can define at least one custom regex rule and scan with
  it through both `scan_text` and `scan_dir`.
- Existing v0.1 usage keeps working without changes.
- Type checkers see the new public config types.
- Docs include one copy-pasteable config example.
- Benchmarks exist and README claims are based on measured output.
- All required workflows pass on the release tag.

## Open Questions

- How much of Betterleaks config should be modeled in v0.2 versus left to
  `config_path`?
- Should the config serializer preserve comments, or is generated TOML enough?
- Should custom rules support validation metadata in v0.2, or wait until v0.3?
- Is Linux arm64 worth adding before PyPI adoption proves demand?
- Should release checksums live as GitHub release assets or in a committed
  manifest generated during release?
